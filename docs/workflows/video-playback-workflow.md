# 视频播放工作流

> 说明：文中的命令耗时和重试节奏用于说明工作流设计取向，属于参考值而非 SLA 或行为保证；真实表现应以当前环境实测为准。

## 概述

用户说"我要看XXX" → 播放视频的完整工作流。

---

## 策略总则

### 平台偏好（播放源选择）

当目标是找到**最佳正版播放源**时，平台偏好明确为：

```
iQiyi → Youku → Tencent → Bilibili
```

说明：
1. 这是**播放源偏好**，不是发现阶段必须逐个平台慢搜的意思
2. 如果上游已经通过结构化结果确认了片名/集数，后续仍应优先落到上述平台顺序中的可用正版链接
3. Bilibili 仍可作为最后备选，尤其适合 UGC 或缺少前三家正版落点的内容

### 发现/搜索顺序

发现阶段与播放源偏好分离：

```
1. 优先走快的结构化/API/metadata 路径（如果有）
2. 需要找正版落点时，再按 iQiyi → Youku → Tencent → Bilibili 做浏览器搜索
3. 当前平台没有结果或无法确认时，才切到下一个平台
```

### 歧义处理

当用户说的词可能有歧义时（如"羊村"可能是喜羊羊也可能是小潮院长的视频）：
1. 优先使用结构化结果确认：标题、作者、集数、时长等
2. 只有在浏览器渲染结果里仍然无法可靠判定时，才截图给用户确认
3. 用户确认后，优先打开**最终目标平台**的 canonical URL；没有直达 URL 时再做 DOM 点击

### 搜索URL模板

```
iQiyi:    https://www.iqiyi.com/so/q_{keyword}
Youku:    https://so.youku.com/search?keyword={keyword}
Tencent:  https://v.qq.com/x/search/?m=key&q={keyword}
Bilibili: https://search.bilibili.com/all?keyword={keyword}
```

## 标准工作流

### Phase 1: 搜索

```
1. 用户: "我要看 XXX"
2. 先尝试结构化/API/metadata 搜索（快路径，如可用）
3. 若已得到可靠候选：先做结果确认，再判断是否已有**最终目标平台**可直接打开的 canonical URL
4. 若没有结构化结果，或需要补正版平台落点：MCP 打开平台搜索页
5. take_snapshot / 解析 DOM
6. 当前平台没结果或无法确认 → 切换到下一个平台重试
```

### Phase 2: 用户确认

```
7. 若结构化结果足够明确 → 直接用标题/作者/集数让用户确认
8. 若浏览器结果仍有歧义 → take_screenshot → 展示给用户
9. 用户确认: "第X个" 或 "不是，换一个"
```

### Phase 3: 点击播放

```
10. 有**最终目标平台**的 canonical URL 时优先直接 open/new_page 到目标页
11. 没有 canonical URL 时，MCP: click (目标视频 UID)
12. 如打开了新页面，MCP: list_pages + select_page
13. MCP: take_snapshot 确认播放器/正文已加载
14. 仅在播放状态仍不明确，或需要用户最终确认时，再 take_screenshot
```

### Phase 4: 善后

```
15. 如果开了多个平台页面 → 关闭不需要的
16. 只保留正在播放的页面
17. 若当前平台失败 → 回到平台偏好链上的下一站继续
```

## 路径与时延

### 两条主要路径

1. **快路径（API / metadata 辅助）**
   - 先拿结构化候选，先做歧义消解
   - 有**最终目标平台**的 canonical URL 时直接打开目标页
   - 通常比纯浏览器路径更快，因为减少了搜索页加载和人工判读

2. **慢路径（纯浏览器兜底）**
   - 打开搜索页
   - snapshot / 解析 / 必要时截图确认
   - 点击结果进入播放页
   - 适用于没有可靠 API、没有直达 URL、或页面结果必须目视确认的场景

### 实测命令耗时

| 命令 | 平均耗时 | 说明 |
|------|----------|------|
| `new_page` | 2.4s | 浏览器搜索的主要开销 |
| `take_snapshot` | 0.8s | DOM 序列化 |
| `take_screenshot` | 0.9s | Windows 截图 |
| `click` | 1.2s | 点击元素 |
| `list_pages` | 0.8s | 列出所有页面 |
| `select_page` | ~0s | 极快 |
| `close_page` | 0.5s | 关闭页面 |

### 优化原则

1. **去掉固定 sleep**：优先靠返回结果和 snapshot 状态推进
2. **先结构化确认，后浏览器打开**：能避免无效页面跳转
3. **最终目标平台 canonical URL 优先**：有直达页就不要先落搜索页
4. **截图后置**：只有在结构化确认不足、DOM 歧义仍存在时才截图
5. **复用 session**：同一个任务内不要重复 initialize

### 错误处理

**页面加载失败**
```
检测到元素 < 50 → 最多重试3次 → 都失败则换平台
```

**点击失败**
```
click返回错误 → 重新snapshot → 重新解析 → 重新点击（最多2次）
```

**视频页加载超时**
```
3次snapshot后还无法确认状态 → 截图展示给用户 → 用户判断
```

## 截图展示规范

### 什么时候截图

| 场景 | 是否截图 | 原因 |
|------|----------|------|
| 搜索结果（结构化信息已足够） | ❌ 通常不需要 | 先走 metadata 确认 |
| 搜索结果（浏览器页仍有歧义） | ✅ 需要 | 用户需要确认 |
| 视频播放页（状态明确） | ❌ 通常不需要 | snapshot 已足够 |
| 视频播放页（状态不明确） | ✅ 需要 | 最终确认 |

### 截图命名

```
video_search_{keyword}_{timestamp}.png   # 搜索结果
video_player_{platform}_{timestamp}.png    # 播放页
video_multi_{timestamp}.png              # 多平台对比
```
